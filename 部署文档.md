# 生图工作站部署文档

## 部署目标

本项目是 Vite 构建的纯前端静态应用。生产构建后只需要部署 `dist/` 目录即可。

由于 API Key 保存在浏览器 IndexedDB，建议部署为个人私有工具，不建议直接开放给不可信公众用户。

GitHub 仓库：<https://github.com/cnxmu/GPT-Image-Web>

## 构建

```bash
npm install
npm run build
```

构建产物位于：

```text
dist/
```

本地预览：

```bash
npm run preview
```

## 静态服务器部署

项目当前按根路径站点部署设计，例如：

```text
https://your-domain.example/
```

项目没有配置 Vite `base` 子路径。生产环境建议使用自有服务器、Nginx、对象存储静态站点、自定义域名静态托管或其他能部署 `dist/` 的服务。

## Nginx 部署示例

示例站点配置：

```nginx
server {
  listen 443 ssl http2;
  server_name example.com;

  root /var/www/image-workbench/dist;
  index index.html;

  ssl_certificate /path/to/fullchain.pem;
  ssl_certificate_key /path/to/privkey.pem;

  include /var/www/image-workbench/deploy/nginx-security-headers.conf;
}
```

项目已提供安全头片段：

```text
deploy/nginx-security-headers.conf
```

该配置包含：

- Content-Security-Policy
- Referrer-Policy
- Permissions-Policy
- X-Content-Type-Options
- X-Frame-Options
- Cross-Origin-Opener-Policy
- SPA fallback：`try_files $uri $uri/ /index.html`

## HTTPS 和访问控制

个人部署时至少启用：

- HTTPS。
- 访问控制，例如 Basic Auth、VPN、Cloudflare Access、Tailscale 或仅内网访问。
- 服务器防火墙，仅开放必要端口。

Nginx Basic Auth 示例：

```nginx
auth_basic "Image Workbench";
auth_basic_user_file /etc/nginx/.htpasswd;
```

生成 `.htpasswd` 可使用：

```bash
sudo apt install apache2-utils
sudo htpasswd -c /etc/nginx/.htpasswd your_name
```

## API Key 策略

当前版本：

- 生图 API Key 和 Agent API Key 由用户在浏览器内输入。
- 密钥保存到当前浏览器 IndexedDB。
- 前端请求 `https://img.xmu.la` 时使用 `Authorization: Bearer <key>`。

注意：

- 不要把 API Key 写进源码。
- 不要把 API Key 写进 `index.html`、静态 JS、Nginx 配置或公开环境变量。
- 共享电脑或不可信浏览器环境不适合保存密钥。

更高安全方案：

- 自建后端代理或边缘函数。
- 真实 API Key 只保存在服务端。
- 前端只调用自己的受保护接口。
- 服务端增加用户认证、限流、审计和额度控制。

## SEO 配置

项目包含基础 SEO：

- `index.html`：标题、描述、关键词、Open Graph、Twitter Card、JSON-LD。
- `public/robots.txt`：引用 sitemap。
- `public/sitemap.xml`：默认收录站点首页。
- `public/manifest.webmanifest`：Web App 名称、描述、图标和主题色。
- `public/og-image.png`：社交平台分享预览图。

当前 `robots.txt` 和 `sitemap.xml` 使用占位域名：

```text
https://example.com/
```

上线前需要同步替换以下文件中的 URL：

- `public/robots.txt`
- `public/sitemap.xml`

如果需要让社交平台稳定抓取分享图，建议在 `index.html` 中把 `og:image` 和 `twitter:image` 从 `/og-image.png` 改为正式域名绝对地址，例如 `https://your-domain.example/og-image.png`。

如果站点是私有工具并加了访问控制，也可以把 `robots.txt` 改为不允许抓取：

```text
User-agent: *
Disallow: /
```

## 数据存储与清理

浏览器 IndexedDB 会保存：

- API Key。
- 设置项。
- 模板。
- 历史记录。
- 图生图参考图资产。
- Agent 对话图片资产。
- 新生成结果图片资产。

应用设置中提供：

- 清除密钥。
- 清空历史与历史图片。
- 清理未引用资源缓存。
- 清空模板。
- 清除全部本地数据。

“清理未引用资源缓存”只删除没有被历史记录或 Agent 对话引用的图片资产，不会破坏仍可回显的历史结果或对话图片。

如果浏览器清除站点数据，上述本地数据会被删除。

## 并发与费用保护

默认生成并发上限为 `20`，可在设置中调整到 `1-100`。

建议：

- 日常使用保持默认 `20`。
- 网络或服务端限流明显时调低。
- 批量生成超过 `20` 张时，应用会弹出确认提示。
- 并发越高，失败率、浏览器内存压力和费用风险越高。

- `dist/` 已部署。

- 访问控制已开启。
- 安全响应头已配置。
- `robots.txt` 和 `sitemap.xml` 的 SEO URL 已替换为正式域名。
- 如需社交平台分享预览，`og:image` / `twitter:image` 已替换为正式域名绝对地址。
- 确认站点没有内置 API Key。
- 在浏览器设置页保存个人 API Key 后再使用。
